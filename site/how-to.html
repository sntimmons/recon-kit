Step 1

Preparing your files

Data Whisperer takes two CSV exports: one from your legacy HRIS and one from your new system. You don't need to reformat or transform anything before uploading. The engine handles column mapping internally.

What columns are required

At minimum, each file needs a worker identifier and a name. The engine matches better the more fields you provide.

ColumnRequired?Used for
worker_idRequiredTier 1 exact identity match
full_nameRequiredName-based matching, Tier 3 / 4
date_of_birthRecommendedTier 2 last4 + DOB matching
hire_dateRecommendedName + hire date matching, mismatch detection
salaryOptionalMismatch detection, correction output
worker_statusOptionalStatus mismatch detection
position / job_titleOptionalJob/org mismatch output
recon_id / pkOptionalSecondary ID matching (Tier 2–3)

Column naming flexibility

Column names don't need to exactly match the list above. The mapping layer recognizes common variations. For example, employee_id, emp_id, and EmployeeNumber are all mapped to worker_id automatically.

If your export uses unusual names, the mapping configuration can be updated to match your schema. Contact us for help with this.

File format

  • Plain CSV (UTF-8 or Windows-1252 encoding)
  • First row must be a header row
  • Dates in YYYY-MM-DD format preferred. Other formats are handled but YYYY-MM-DD is safest.
  • Salary as a plain number (no currency symbols or commas)
  • No requirement for a specific row count. Tested from dozens to hundreds of thousands of rows.
Export tip: Pull the most complete export your system allows. More fields means better mismatch detection. You can redact sensitive fields like SSN/NI after confirming matching works, but include them for the initial run if you can.
Step 2

Running the tool

Once your files are ready, the process is straightforward.

Using the web upload (coming soon)

  1. Go to the Tool page. You'll see two upload zones side by side.

  2. Upload your legacy export to the left zone and your new system export to the right.

  3. Click Run Reconciliation. The engine runs locally in your browser. Nothing is sent to any server.

  4. Wait for the run to complete. Progress updates appear in real time. A full run on 50,000 employees typically takes under 30 seconds.

Using the demo

If you want to see the output before using your own data, the Demo page runs with preloaded sample data. No upload required. The results show a realistic output for a 250-employee migration scenario.

Important: Data Whisperer is a read-only analysis tool. It never modifies your source files. The correction output files it generates are separate, new files you can choose to import into your target system.
Output

Reading the match summary

The match summary is the first thing you see after a run completes. Here's what every field means.

Top-level stats

FieldWhat it means
Legacy recordsTotal rows in your legacy system export.
New system recordsTotal rows in your new system export.
Auto-matchedPairs resolved with enough confidence to proceed without human review.
Review requiredRecords the engine couldn't resolve confidently. Routed to the review queue.
Match rateAuto-matched ÷ Legacy records. Above 95% is typical for clean migrations.
Field mismatchesTotal field-level discrepancies found across all matched pairs.

Mismatch breakdown

Mismatches are grouped into five categories. Each row shows how many pairs have that type of discrepancy, what percentage of matched pairs that represents, how many were auto-approved (within policy thresholds), and how many need human review.

  • Salary: The annual base salary differs between the legacy and new system record.
  • Hire date: The hire date recorded in the two systems doesn't match.
  • Job / Org: Position, department, location, or organizational unit discrepancy.
  • Status: Worker status (Active, Inactive, Leave, etc.) differs between systems.
  • Pay rate: The hourly or periodic pay rate differs from what the new system holds.
Engine detail

Match strategies explained

The engine applies six matching strategies in priority order. Each strategy is tried in sequence. Once a record is matched and claimed, it's no longer available for lower-tier strategies. This is called staged 1-to-1 resolution, and it prevents one record from being matched to multiple counterparts.

TierStrategyHow it worksConfidence
T1worker_idExact match on the employee identifier field in both systems.Highest
T2recon_idMatch on a secondary reconciliation identifier, if present.High
T2last4 + DOBLast 4 digits of SSN/NI number combined with date of birth.High
T3DOB + nameDate of birth combined with normalized full name.Medium
T3name + hire dateNormalized full name combined with hire date.Medium
T4name similarityFuzzy name matching using sequence similarity scoring. Used as a last resort.Lower
Name normalization: Before matching on name, the engine normalizes both sides: lowercasing, stripping accents, removing punctuation, and standardizing whitespace. "José García-López" and "jose garcia lopez" match correctly.
Output

The review queue

Not every record can be matched automatically with confidence. Records that don't resolve cleanly are written to a review queue for human assessment.

What ends up in the review queue

  • Records where no match was found in any tier
  • Records where the match strategy was low-confidence (name similarity only)
  • Matched pairs where a field mismatch exceeds the policy confidence threshold
  • Pairs where the worker ID exists in one system but not the other

How to work through the review queue

  1. Open review_queue.csv from the run output folder.

  2. Check the reason column. It explains exactly why the record was flagged.

  3. Compare the old_* and new_* fields side by side for the flagged record.

  4. Decide: confirm the match, identify the correct record, or mark as no-match if the employee genuinely doesn't exist in the target system.

  5. Corrections to review items can be made directly in the correction CSV before importing.

Target: A well-prepared migration should have a review queue of under 5% of total records. Consistently above 10% usually signals a data quality problem worth investigating before go-live.
Output

Audit files

Every run produces a complete set of audit outputs. These are designed to support a formal go-live sign-off. You can hand these directly to a project manager or auditor.

What gets audited

The audit suite runs 15 query checks (Q0–Q15) across the matched pairs. These cover:

  • Duplicate detection: are any worker IDs matched more than once?
  • Orphaned records: legacy records with no new-system match
  • Salary distribution analysis: salary buckets, outliers, and percentage changes
  • Hire date distribution: expected vs suspicious default dates
  • Status transition patterns: unexpected Active to Inactive changes
  • Field completeness: which fields are blank, null, or default values

Audit CSV files

Each audit query that finds results produces a CSV in the audit/ subfolder of the run output. These are named audit_q1_*.csv through audit_q15_*.csv. Each file contains the full rows that triggered that query, plus annotations.

The audit workbook

recon_workbook.xlsx consolidates all audit output into a single Excel workbook with eight tabs: Summary, All Matches, Salary Mismatches, Status Mismatches, Hire Date Mismatches, Job/Org Mismatches, Review Queue, and Corrections Manifest. This is the file you share with stakeholders.

Quality control

The sanity gate

The sanity gate is an automated policy check that evaluates your reconciliation results against configurable thresholds before generating correction files. It's designed to catch situations where the data is too broken to trust.

What it checks

CheckDefault thresholdWhat it means
Salary mismatch rateUnder 40%More than 40% of employees with salary discrepancies suggests a systemic data error.
Hire date default rateUnder 5%Dates that look like system defaults (e.g. 2026-02-01) are suspicious at scale.
Status change rateUnder 10%More than 1 in 10 employees changing status during migration is a data integrity signal.
Unmatched recordsConfigurableA hard limit on how many records can remain unmatched before the run is blocked.

Gate outcomes

Gate PASSED: All thresholds are within policy. The engine proceeds to generate correction files. The full output is available.

Gate FAILED: One or more thresholds were breached. Correction file generation is blocked. You should not apply corrections to a system with data at this level of inconsistency. The audit workbook, review queue, and all analysis outputs are still generated so you can investigate.

Gate failed? Look at the sanity report first. It will tell you exactly which threshold was breached and by how much. Common causes: a hire date default batch migration, a system export that omitted a field, or a mapping error where salary was pulled from the wrong column.
Output

Correction files

When the sanity gate passes, Data Whisperer generates ready-to-import correction files for each mismatch category. These are formatted for Workday, with the columns your HRIS team needs.

Files generated

  • corrections_salary.csv: worker_id, effective_date, compensation_amount, currency, reason
  • corrections_status.csv: worker_id, effective_date, worker_status, reason
  • corrections_hire_date.csv: worker_id, hire_date, reason
  • corrections_job_org.csv: worker_id, position, district, location, reason
  • corrections_manifest.csv: master list of all correction records with action (APPROVE / REVIEW) and confidence score

APPROVE vs REVIEW

Every correction record is tagged with an action. APPROVE means the engine assessed the mismatch, applied the confidence policy, and determined the correction is safe to apply automatically. REVIEW means a human should look at the record before importing.

The confidence thresholds are set conservatively by default. Salary corrections require 97% confidence. Hire date corrections require 95%. Status changes require 98%. You can adjust these in the policy configuration.

Before importing: Always do a spot-check on 10–20 rows from the APPROVE set before running a mass import. The correction files are designed to be safe, but no automated system replaces a final human sanity check.

Common questions

FAQ

Does Data Whisperer connect to my HRIS system directly?

No. It works exclusively with CSV exports. There's no API integration or direct system access. You export your data, run the reconciliation, and import the corrections manually. This is intentional. It keeps the tool simple and data sovereign.

Does my data leave my machine?

No. The engine runs entirely locally. Your CSV files are processed in your browser or on your local machine. Nothing is uploaded to any server. There is no cloud backend.

My match rate is below 90%. Is that a problem?

It depends. A match rate below 90% usually means one of three things: (1) the two systems use different worker ID schemes and the fallback strategies can't bridge the gap, (2) there are genuine population differences between the legacy and new system that need investigating, or (3) there's a data quality issue in one of the exports. Look at the unmatched records in the review queue. The reason codes will point you to the cause.

Can I run this against a subset of employees?

Yes. Filter your exports before uploading. The engine doesn't care about total population size. Running against a department or cost center first is a good way to validate your configuration before processing the full workforce.

How do I know which match strategy found each pair?

Every matched pair in the output has a match_source column. It contains a value like worker_id, last4_dob, or name_hire_date, directly identifying the tier that resolved the match. Lower-tier matches are worth reviewing even if they auto-approved.

Can I use this for a non-Workday migration?

Yes. The engine is system-agnostic. It works with any two HR datasets in CSV format. The correction file format defaults to Workday column conventions, but the underlying analysis and match output work for any target system. Contact us if you need a different correction format.
